# `@qkb/contracts` — Solidity for Qualified Key Binding (Phase 1)

This file is the orientation doc for any future contributor (human or
agent) working inside `packages/contracts/`. It is intentionally
opinionated and prescriptive: follow the patterns laid down here unless
you have a specific reason and a green test suite to back you up.

## 1. Package purpose

QKB Phase 1's on-chain surface. Two production contracts plus one test
helper:

| File | Role |
|---|---|
| `src/QKBVerifier.sol`            | Library wrapping the snarkjs Groth16 verifier. Owns the public-input packing layout and the secp256k1 pubkey → Ethereum address derivation. |
| `src/QKBRegistry.sol`            | Reference register-then-authenticate registry. Holds bindings, dispatches between RSA and ECDSA Groth16 verifiers, exposes `isActiveAt` for downstream callers, and admin-rotates the trusted-list root. |
| `src/verifier/StubGroth16Verifier.sol` | **Test-only** stand-in. Returns a configurable bool. Never deploy. |

Real `QKBGroth16Verifier` Solidity files arrive from `@qkb/circuits`
(one per algorithm variant) and live under `src/verifier/` once
delivered. They are auto-generated by snarkjs and committed unmodified.

The user-facing pattern is the spec's:

1. Browser proves QKB knowledge of `(QES, pk)` → Groth16 proof + 13 public signals.
2. User submits proof to `QKBRegistry.register(...)` once.
3. Future authentication = standard `ecrecover(sig)` + `registry.isActiveAt(pkAddr, t)`.

## 2. How to run

The repo-root `foundry.toml` is THE Foundry configuration. **Do not add
a per-package `foundry.toml`.** It already wires `src`, `test`, `script`,
optimizer, `via_ir`, `solc = 0.8.24`, and an `fs_permissions` entry for
`./packages/contracts/test/fixtures`.

All commands are safe to run from the repo root or from
`packages/contracts/` (the package's `package.json` invokes `forge` with
`--root ../..`).

```bash
# Build
forge build

# Test (fast)
forge test

# Test (verbose, full traces)
forge test -vvv

# Run a single suite / test
forge test --match-path 'packages/contracts/test/QKBRegistry.register.t.sol'
forge test --match-test test_register_rsa_happyPath_writesBindingAndEmits

# Gas snapshot — refresh
forge snapshot --snap packages/contracts/snapshots/gas-snapshot.txt

# Gas snapshot — CI guard
forge snapshot --check --snap packages/contracts/snapshots/gas-snapshot.txt

# Local deploy dry-run against an anvil fork
anvil --port 8545 &
ROOT_TL=0x0000…01 \
REGISTRY_ADMIN=0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266 \
  forge script packages/contracts/script/Deploy.s.sol \
    --fork-url http://localhost:8545
```

The `ci` profile (`FOUNDRY_PROFILE=ci`) raises verbosity and bumps fuzz
runs to 256.

## 3. On-chain architecture

```
                  ┌──────────────────────────────┐
                  │   QKBGroth16Verifier (RSA)   │  ← snarkjs-generated
                  └──────────────┬───────────────┘
                                 │
                                 ▼
                          IGroth16Verifier
                                 ▲
                                 │
                  ┌──────────────┴───────────────┐
                  │  QKBGroth16Verifier (ECDSA)  │  ← snarkjs-generated
                  └──────────────────────────────┘

                  ┌──────────────────────────────┐
                  │       QKBVerifier (lib)      │  packs Inputs → uint256[13],
                  │  declHash whitelist + pk→addr│  enforces declHash whitelist
                  └──────────────┬───────────────┘
                                 │
                                 ▼
                  ┌──────────────────────────────┐
                  │       QKBRegistry            │  storage, register/expire/
                  │   (rsaVerifier, ecdsaVerifier│  isActiveAt, admin, dispatch
                  │    + admin + bindings)       │
                  └──────────────────────────────┘
```

**Dispatch model** (orchestration §2.0): each registered binding declares
which Groth16 circuit produced its proof via `algorithmTag`:

- `algorithmTag = 0` — RSA-PKCS#1 v1.5 2048
- `algorithmTag = 1` — ECDSA P-256
- anything else — `revert UnknownAlgorithm()`

The registry holds two **settable** verifier addresses (`rsaVerifier`,
`ecdsaVerifier`); admin can rotate either when a fresh ceremony ships a
new circuit.

`StubGroth16Verifier` is a one-storage-slot mock used pervasively in unit
tests so each individual contract behaviour can be exercised without
touching an actual Groth16 verifier. It is replaced by the real
generated verifier in the integration suite (Task 10, pending circuits
delivery).

## 4. Interface contracts

These layouts are FROZEN by orchestration §2.0 / §2.2 / spec §6.1 and
shared with `@qkb/circuits` (witness packer) and `@qkb/web` (proof
submitter). Touching the layout requires coordinated changes across all
three packages plus a new ceremony.

### 4.1 `QKBVerifier.Inputs`

```solidity
struct Inputs {
    uint256[4] pkX;          // little-endian limbs: limb[0] = lowest order
    uint256[4] pkY;
    bytes32    ctxHash;      // Poseidon(ctx) or canonical empty-bytes constant
    bytes32    rTL;          // Poseidon Merkle root from LOTL flatten
    bytes32    declHash;     // SHA-256 of EN or UK canonical declaration text
    uint64     timestamp;    // Unix seconds, unsigned
    uint8      algorithmTag; // 0 = RSA, 1 = ECDSA (orchestration §2.0)
}
```

### 4.2 Public-signal array (`uint256[13]`)

`QKBVerifier.verify` packs `Inputs` into the 13-element public-signal
array the snarkjs Groth16 verifier expects:

| idx     | source             |
|---------|--------------------|
| 0..3    | `i.pkX[0..3]`      |
| 4..7    | `i.pkY[0..3]`      |
| 8       | `uint256(i.ctxHash)` |
| 9       | `uint256(i.rTL)`   |
| 10      | `uint256(i.declHash)` |
| 11      | `uint256(i.timestamp)` |
| 12      | `uint256(i.algorithmTag)` |

If you find yourself writing `arr[12] = 0` again — STOP. That field
was `reserved` in an earlier draft of the spec and is now load-bearing.

### 4.3 secp256k1 limb encoding

`pkX` and `pkY` are 4 × uint64 limbs in **little-endian** order. limb[0]
is the lowest-order 64 bits. To recover the integer value:

```
x = limb[0] | (limb[1] << 64) | (limb[2] << 128) | (limb[3] << 192)
```

`bytes32(x)` is already big-endian on EVM — there is **no** byte-reversal
step. The Ethereum address is `keccak256(bytes32(x) || bytes32(y))[12:]`.
This is verified against `vm.addr(priv)` for 18 pinned (priv, x, y)
vectors in `test/QKBVerifier.fuzz.t.sol`; if you ever change the packing,
re-run that suite first.

## 5. Declaration digest pinning

`src/constants/DeclarationHashes.sol` pins:

```
EN = 0xf83a393242585b93dea1474fbfc92a06dd90629f3e159b0d230548274315c89b
UK = 0x692d0666a14c9e70ebadbfb864a18e152a70be061924902423e52b4ae21492b7
```

These MUST equal the digests recorded in `fixtures/declarations/digests.json`
at repo root. The values also appear in:

- `circuits/binding/DeclarationWhitelist.circom`
- The web app's i18n declaration strings (computed at build time)

Drift in any of these three breaks the user flow silently. To enforce
that, `test/Declarations.t.sol` reads the canonical text bytes mirrored
under `test/fixtures/declarations/` (only that path is allowed by
`fs_permissions`) and asserts `sha256(text) == DeclarationHashes.X`.

If a declaration ever needs to change:

1. Update `fixtures/declarations/{en,uk}.txt` at repo root.
2. Mirror to `packages/contracts/test/fixtures/declarations/`.
3. Update `fixtures/declarations/digests.json`.
4. Update `DeclarationHashes.{EN,UK}`.
5. Update the circuit and web copies.
6. Re-run all three test suites.

The drift-detector test will fail loudly if you miss step 4.

## 6. Error taxonomy

All `QKBRegistry` errors are custom errors (cheaper, structured ABI):

| Error              | Where it fires                                                       |
|--------------------|----------------------------------------------------------------------|
| `RootMismatch()`         | `i.rTL != trustedListRoot` in `register`                       |
| `AlreadyBound()`         | `bindings[pkAddr].status != NONE` in `register`                |
| `BindingTooOld()`        | `block.timestamp > i.timestamp + MAX_AGE` (90 days)            |
| `BindingFromFuture()`    | `i.timestamp > block.timestamp`                                |
| `InvalidProof()`         | `QKBVerifier.verify(...)` returns false (covers bad declHash too) |
| `UnknownAlgorithm()`     | `i.algorithmTag` is neither 0 nor 1                            |
| `NotBound()`             | `expire` called for a never-bound or already-expired key       |
| `BadExpireSig()`         | `ecrecover(expireDigest, sig)` ≠ `pkAddr`                      |
| `NotAdmin()`             | non-admin calls an admin-only function                         |
| `ZeroAddress()`          | constructor or admin setter receives `address(0)`              |

Note on `NotBound` doubling for already-expired bindings: documented in
`test/QKBRegistry.expire.t.sol`. Rationale — once a binding leaves
`ACTIVE`, there is nothing to expire; reusing `NotBound` keeps the ABI
minimal.

Note on `InvalidProof` doubling for unknown declaration hashes: the
`declHash ∈ {EN, UK}` gate lives inside `QKBVerifier.verify()` for
defence-in-depth, so an unknown declHash short-circuits there and the
caller-facing error is `InvalidProof`. Documented in
`test/QKBRegistry.register.t.sol`.

`VerifierUpdated(uint8 indexed algorithmTag, address oldVerifier, address newVerifier)`
is an event, not an error; it fires from the constructor (genesis) and
from `setRsaVerifier` / `setEcdsaVerifier`.

## 7. Tests

Each public/external function of `QKBRegistry` has a happy-path test AND
at least one revert test. Mirror this pattern when adding new behaviour.

| File | Coverage |
|---|---|
| `test/Smoke.t.sol`                       | Trivial scaffold sanity check.                                                                  |
| `test/Declarations.t.sol`                | DeclarationHashes.{EN,UK} match canonical files; `isAllowed` accepts both, rejects others.       |
| `test/QKBVerifier.t.sol`                 | `verify` packs inputs and respects declHash whitelist + stub accept/reject; toPkAddress vs G/2G/3G. |
| `test/QKBVerifier.fuzz.t.sol`            | toPkAddress vs `vm.addr(priv)` over 18 pinned secp256k1 vectors (true Sol-side fuzz isn't possible — see §10). |
| `test/QKBRegistry.admin.t.sol`           | Constructor (initial state, zero-address rejection × 3), root rotation, admin transfer, both verifier setters.  |
| `test/QKBRegistry.register.t.sol`        | Happy path × 2 (RSA + ECDSA dispatch), 7 revert paths (proof, root, future, age, double, declHash, unknown alg), cross-algorithm dispatch sanity (RSA accepts but ECDSA route rejects, vice versa). |
| `test/QKBRegistry.expire.t.sol`          | Happy path, wrong signer, tampered digest, never-bound, double-expire.                          |
| `test/QKBRegistry.isActiveAt.t.sol`      | Full truth table: NONE, pre-bind, ACTIVE-now, ACTIVE-far-future, EXPIRED-before/at/after expiredAt. |
| `test/QKBRegistry.integration.t.sol`     | **Pending Task 10** — real Groth16 verifier + real proof fixture.                                 |
| `test/helpers/SignatureHelpers.sol`      | `expireDigest(pkAddr, chainId, boundAt)` — single source of truth shared with off-chain fixture generators. |

When adding a feature: write the test first. Stub-verifier tests are
fast and exhaustive; reserve the integration suite for end-to-end
exercises with real proofs. If you introduce a new error, add one
revert test per call site that can fire it.

## 8. Gas

Current baseline (snapshots/gas-snapshot.txt):

- `register` (happy path, RSA dispatch): ~120k — well under the 400k spec budget.
- `expire` (happy path): ~50k for the diff over a fresh `register`.
- `isActiveAt`: single SLOAD plus arithmetic, view function.
- Both admin setters: ~28k for state change + event.
- Constructor: ~3.7M (generated verifier code dominates once real ones land).

CI MUST run `forge snapshot --check --snap packages/contracts/snapshots/gas-snapshot.txt`
to flag regressions. Refresh deliberately — when you make an intentional
gas-cost change, re-run `forge snapshot --snap …` and call out the delta
in the commit message.

## 9. Deploy

### 9.1 Credentials live in `.env`

Both deploy scripts read admin credentials from environment variables.
Forge auto-loads `.env` from the repo root (the foundry config root).
The real `.env` is **gitignored** (root `.gitignore`, with a
`!.env.example` whitelist for the template) and MUST NEVER appear in:

- commits (Git history is forever, and a leaked key cannot be unleaked),
- tests (use `vm.envXxx` at runtime; never bake values into source),
- fixtures under `test/fixtures/`,
- CI workflow YAML (use the CI provider's secret store; never echo).

If a key is ever transmitted over a non-secure channel (Slack, chat,
issue comment, agent message), treat it as compromised and rotate
before any production use.

Required env for `Deploy.s.sol`:

| Var                  | Type    | Notes                                                                |
|----------------------|---------|----------------------------------------------------------------------|
| `ROOT_TL`            | bytes32 | Initial `trustedListRoot`.                                           |
| `ADMIN_PRIVATE_KEY`  | uint256 | Broadcasting key. Also the registry admin.                           |
| `ADMIN_ADDRESS`      | address | Must equal `vm.addr(ADMIN_PRIVATE_KEY)`. Script reverts `AdminMismatch` otherwise — catches the easy "pasted one but not the other" mistake. |
| `RSA_VERIFIER_ADDR`  | address | Optional. If absent or `0x0`, deploys a `StubGroth16Verifier`.       |
| `ECDSA_VERIFIER_ADDR`| address | Same convention.                                                     |

Required env for `UpdateTrustedRoot.s.sol`:

| Var                  | Type    | Notes                                                                |
|----------------------|---------|----------------------------------------------------------------------|
| `REGISTRY_ADDR`      | address | Deployed `QKBRegistry`.                                              |
| `NEW_ROOT_TL`        | bytes32 | New `trustedListRoot`.                                               |
| `ADMIN_PRIVATE_KEY`  | uint256 | Broadcasting key; must match `registry.admin()` or call reverts on-chain `NotAdmin`. |

## 10. Gotchas

- **EVM word is big-endian.** `bytes32(uint256)` already gives the
  big-endian byte representation of the integer. Do NOT add a
  reverse-bytes step in `toPkAddress` — earlier draft pseudocode had
  one, it's wrong, and an entire test class (`QKBVerifier.fuzz.t.sol`)
  exists to keep that mistake from being re-introduced.
- **`vm.addr(priv)` does not expose the underlying secp256k1 (x, y).**
  Solidity has no native curve op to derive xy from a fuzzed privkey.
  We use 18 pinned (priv, x, y) vectors generated with python's `ecdsa`
  library. For a true fuzz you'd FFI to a Rust secp256k1 helper —
  acceptable but not currently wired.
- **`forge script` writes a `broadcast/` directory.** It writes to both
  `./broadcast/` (worktree root) and `packages/contracts/broadcast/`
  depending on invocation. Both are gitignored at repo root; if you see
  one of them appear, double-check before staging.
- **`ecrecover` is used in two places.** `QKBVerifier.toPkAddress`
  computes `keccak256(x||y)[12:]` directly from the limb-packed pubkey
  (no signature involved); `QKBRegistry.expire` uses real `ecrecover`
  on a domain-separated digest (`keccak256(abi.encode("QKB_EXPIRE_V1",
  pkAddr, block.chainid, boundAt))`).
- **Stub verifier is test-only.** Never deploy `StubGroth16Verifier` to
  a chain users will rely on. The Deploy script's stub fallback exists
  solely for anvil dry-runs and CI smoke tests.
- **Submodules.** `lib/forge-std` (v1.9.4) and `lib/openzeppelin-contracts`
  (v5.1.0) are git submodules pinned in `foundry.lock` at repo root.
  Run `git submodule update --init --recursive` after a fresh checkout.
- **`fs_permissions` is restrictive on purpose.** Only
  `./packages/contracts/test/fixtures` is readable from Solidity. If you
  need a new fixture path, mirror the file under that directory rather
  than widening the permission scope.

## 11. Extending

### Adding a third QES algorithm (e.g. EdDSA, DSTU-4145)

1. Pick a new `algorithmTag` constant — append, never reuse:
   ```solidity
   uint8 internal constant ALG_EDDSA = 2;
   ```
2. Add a verifier slot + setter:
   ```solidity
   IGroth16Verifier public eddsaVerifier;
   function setEddsaVerifier(IGroth16Verifier v) external onlyAdmin { … }
   ```
3. Extend the constructor to take and validate the new verifier.
4. Add a dispatch arm in `register`:
   ```solidity
   else if (i.algorithmTag == ALG_EDDSA) v = eddsaVerifier;
   ```
5. Update `Binding` and `BindingRegistered` consumers if needed (the
   `algorithmTag` field already supports up to 255 algorithms).
6. Add admin tests (zero-address constructor revert, setter rotate +
   onlyAdmin + zero), happy path and dispatch-mismatch revert tests in
   `test/QKBRegistry.register.t.sol`.
7. Refresh the gas snapshot.
8. Update this CLAUDE.md and orchestration §2.0.

### Adding a new circuit-level public input

Touches all three packages; coordinate with circuits-eng and web-eng
before starting. `arr[12]` is fully consumed; an additional input
requires the snarkjs verifier to expand to `uint256[14]`, the
`IGroth16Verifier` interface to widen, every test to update, and the
ceremony to re-run.

## 12. Phase 1 explicit non-goals

For clarity when reviewing scope-creep PRs — these are deferred to
Phase 2 or later, NOT lurking bugs:

- **Historical-root rotation receipts.** `updateTrustedListRoot` overwrites
  in place; there is no on-chain log of the prior roots beyond the event
  stream. Indexers reconstruct history off-chain.
- **Per-context binding enforcement.** `ctxHash` is stored but the
  registry does not gate `isActiveAt` on a matching context. Consumers
  that care about context bind via the off-chain binding statement.
- **QEAA attribute disclosure.** The proof discloses pk, ctx, declaration,
  algorithm, timestamp, and the LOTL root inclusion — nothing else from
  the QES certificate. Selective attribute disclosure is Phase 2.
- **Escrow hooks (QIE).** The `escrow_commitment` field in the binding
  statement is `null` in Phase 1; on-chain escrow contracts are Phase 2.
- **Mixed RSA/ECDSA chains.** The proof attests to a single QES leaf
  cert chained to a single intermediate, both signed with the same
  algorithm. Mixed-algorithm chains are out of scope.
- **EIP-712 typed-data signatures for `expire`.** The current digest is
  a raw `keccak256(abi.encode(domain, …))` — sufficient because the
  bound key is a one-purpose key, but future versions may switch to
  EIP-712 for wallet UX. Bump `EXPIRE_DOMAIN` if you do.
- **Account abstraction integration.** `register` requires an EOA-style
  caller (anyone can submit, but the proof contents identify the bound
  key). Smart-contract-account wrapper helpers are Phase 2.
