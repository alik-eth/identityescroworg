# `@zkqes/contracts` — zkqes Solidity contracts (verifier library + registry)

> **zkqes V3/V4/V5/V5.1/V5.2 combined.** The earlier sections below
> document the register-then-authenticate binding registry and its verifier
> wrapper. Section 13 covers the V3/V4 escrow additions: the
> 14-signal public layout, dual-verifier dispatch, nullifier primitive,
> escrow surface, and arbitrator event contract. Section 14 covers V5.

This file is the orientation doc for any future contributor (human or
agent) working inside `packages/contracts/`. It is intentionally
opinionated and prescriptive: follow the patterns laid down here unless
you have a specific reason and a green test suite to back you up.

## 1. Package purpose

zkqes on-chain surface. Two production contracts plus one test
helper:

| File | Role |
|---|---|
| `src/ZkqesVerifier.sol`            | Library wrapping the snarkjs Groth16 verifier. Owns the public-input packing layout and the secp256k1 pubkey → Ethereum address derivation. |
| `src/ZkqesRegistry.sol`            | Reference register-then-authenticate registry. Holds bindings, dispatches between RSA and ECDSA Groth16 verifiers, exposes `isActiveAt` for downstream callers, and admin-rotates the trusted-list root. |
| `src/verifier/StubGroth16Verifier.sol` | **Test-only** stand-in. Returns a configurable bool. Never deploy. |

Real `ZkqesGroth16Verifier` Solidity files arrive from `@zkqes/circuits`
(one per algorithm variant) and live under `src/verifier/` once
delivered. They are auto-generated by snarkjs and committed unmodified.

The user-facing pattern is the spec's:

1. Browser proves zkqes knowledge of `(QES, pk)` → Groth16 proof + 13 public signals.
2. User submits proof to `ZkqesRegistry.register(...)` once.
3. Future authentication = standard `ecrecover(sig)` + `registry.isActiveAt(pkAddr, t)`.

## 2. How to run

The repo-root `foundry.toml` is THE Foundry configuration. **Do not add
a per-package `foundry.toml`.** It already wires `src`, `test`, `script`,
optimizer, `via_ir`, `solc = 0.8.24`, and an `fs_permissions` entry for
`./packages/contracts/test/fixtures`.

All commands are safe to run from the repo root or from
`packages/contracts/` (the package's `package.json` invokes `forge` with
`--root ../..`).


## 3. On-chain architecture

```
                  ┌──────────────────────────────┐
                  │   ZkqesGroth16Verifier (RSA)   │  ← snarkjs-generated
                  └──────────────┬───────────────┘
                                 │
                                 ▼
                          IGroth16Verifier
                                 ▲
                                 │
                  ┌──────────────┴───────────────┐
                  │  ZkqesGroth16Verifier (ECDSA)  │  ← snarkjs-generated
                  └──────────────────────────────┘

                  ┌──────────────────────────────┐
                  │       ZkqesVerifier (lib)      │  packs Inputs → uint256[13],
                  │  declHash whitelist + pk→addr│  enforces declHash whitelist
                  └──────────────┬───────────────┘
                                 │
                                 ▼
                  ┌──────────────────────────────┐
                  │       ZkqesRegistry            │  storage, register/expire/
                  │   (rsaVerifier, ecdsaVerifier│  isActiveAt, admin, dispatch
                  │    + admin + bindings)       │
                  └──────────────────────────────┘
```

**Dispatch model** (orchestration §2.0): each registered binding declares
which Groth16 circuit produced its proof via `algorithmTag`:

- `algorithmTag = 0` — RSA-PKCS#1 v1.5 2048
- `algorithmTag = 1` — ECDSA P-256
- anything else — `revert UnknownAlgorithm()`

The registry holds two **settable** verifier addresses (`rsaVerifier`,
`ecdsaVerifier`); admin can rotate either when a fresh ceremony ships a
new circuit.

`StubGroth16Verifier` is a one-storage-slot mock used pervasively in unit
tests so each individual contract behaviour can be exercised without
touching an actual Groth16 verifier. It is replaced by the real
generated verifier in the integration suite (Task 10, pending circuits
delivery).

## 4. Interface contracts

These layouts are FROZEN by orchestration §2.0 / §2.2 / spec §6.1 and
shared with `@zkqes/circuits` (witness packer) and `@zkqes/web` (proof
submitter). Touching the layout requires coordinated changes across all
three packages plus a new ceremony.

### 4.1 `ZkqesVerifier.Inputs`

```solidity
struct Inputs {
    uint256[4] pkX;          // little-endian limbs: limb[0] = lowest order
    uint256[4] pkY;
    bytes32    ctxHash;      // Poseidon(ctx) or canonical empty-bytes constant
    bytes32    rTL;          // Poseidon Merkle root from LOTL flatten
    bytes32    declHash;     // SHA-256 of EN or UK canonical declaration text
    uint64     timestamp;    // Unix seconds, unsigned
    uint8      algorithmTag; // 0 = RSA, 1 = ECDSA (orchestration §2.0)
}
```

### 4.2 Public-signal array (`uint256[13]`)

`ZkqesVerifier.verify` packs `Inputs` into the 13-element public-signal
array the snarkjs Groth16 verifier expects:

| idx     | source             |
|---------|--------------------|
| 0..3    | `i.pkX[0..3]`      |
| 4..7    | `i.pkY[0..3]`      |
| 8       | `uint256(i.ctxHash)` |
| 9       | `uint256(i.rTL)`   |
| 10      | `uint256(i.declHash)` |
| 11      | `uint256(i.timestamp)` |
| 12      | `uint256(i.algorithmTag)` |

If you find yourself writing `arr[12] = 0` again — STOP. That field
was `reserved` in an earlier draft of the spec and is now load-bearing.

### 4.3 secp256k1 limb encoding

`pkX` and `pkY` are 4 × uint64 limbs in **little-endian** order. limb[0]
is the lowest-order 64 bits. To recover the integer value:

```
x = limb[0] | (limb[1] << 64) | (limb[2] << 128) | (limb[3] << 192)
```

`bytes32(x)` is already big-endian on EVM — there is **no** byte-reversal
step. The Ethereum address is `keccak256(bytes32(x) || bytes32(y))[12:]`.
This is verified against `vm.addr(priv)` for 18 pinned (priv, x, y)
vectors in `test/ZkqesVerifier.fuzz.t.sol`; if you ever change the packing,
re-run that suite first.

## 5. Declaration digest pinning

`src/constants/DeclarationHashes.sol` pins:

```
EN = 0xf83a393242585b93dea1474fbfc92a06dd90629f3e159b0d230548274315c89b
UK = 0x692d0666a14c9e70ebadbfb864a18e152a70be061924902423e52b4ae21492b7
```

These MUST equal the digests recorded in `fixtures/declarations/digests.json`
at repo root. The values also appear in:

- `circuits/binding/DeclarationWhitelist.circom`
- The web app's i18n declaration strings (computed at build time)

Drift in any of these three breaks the user flow silently. To enforce
that, `test/Declarations.t.sol` reads the canonical text bytes mirrored
under `test/fixtures/declarations/` (only that path is allowed by
`fs_permissions`) and asserts `sha256(text) == DeclarationHashes.X`.

If a declaration ever needs to change:

1. Update `fixtures/declarations/{en,uk}.txt` at repo root.
2. Mirror to `packages/contracts/test/fixtures/declarations/`.
3. Update `fixtures/declarations/digests.json`.
4. Update `DeclarationHashes.{EN,UK}`.
5. Update the circuit and web copies.
6. Re-run all three test suites.

The drift-detector test will fail loudly if you miss step 4.

## 6. Error taxonomy

All `ZkqesRegistry` errors are custom errors (cheaper, structured ABI):

| Error              | Where it fires                                                       |
|--------------------|----------------------------------------------------------------------|
| `RootMismatch()`         | `i.rTL != trustedListRoot` in `register`                       |
| `AlreadyBound()`         | `bindings[pkAddr].status != NONE` in `register`                |
| `BindingTooOld()`        | `block.timestamp > i.timestamp + MAX_AGE` (90 days)            |
| `BindingFromFuture()`    | `i.timestamp > block.timestamp`                                |
| `InvalidProof()`         | `ZkqesVerifier.verify(...)` returns false (covers bad declHash too) |
| `UnknownAlgorithm()`     | `i.algorithmTag` is neither 0 nor 1                            |
| `NotBound()`             | `expire` called for a never-bound or already-expired key       |
| `BadExpireSig()`         | `ecrecover(expireDigest, sig)` ≠ `pkAddr`                      |
| `NotAdmin()`             | non-admin calls an admin-only function                         |
| `ZeroAddress()`          | constructor or admin setter receives `address(0)`              |

Note on `NotBound` doubling for already-expired bindings: documented in
`test/ZkqesRegistry.expire.t.sol`. Rationale — once a binding leaves
`ACTIVE`, there is nothing to expire; reusing `NotBound` keeps the ABI
minimal.

Note on `InvalidProof` doubling for unknown declaration hashes: the
`declHash ∈ {EN, UK}` gate lives inside `ZkqesVerifier.verify()` for
defence-in-depth, so an unknown declHash short-circuits there and the
caller-facing error is `InvalidProof`. Documented in
`test/ZkqesRegistry.register.t.sol`.

`VerifierUpdated(uint8 indexed algorithmTag, address oldVerifier, address newVerifier)`
is an event, not an error; it fires from the constructor (genesis) and
from `setRsaVerifier` / `setEcdsaVerifier`.

## 7. Tests

Each public/external function of `ZkqesRegistry` has a happy-path test AND
at least one revert test. Mirror this pattern when adding new behaviour.

| File | Coverage |
|---|---|
| `test/Smoke.t.sol`                       | Trivial scaffold sanity check.                                                                  |
| `test/Declarations.t.sol`                | DeclarationHashes.{EN,UK} match canonical files; `isAllowed` accepts both, rejects others.       |
| `test/ZkqesVerifier.t.sol`                 | `verify` packs inputs and respects declHash whitelist + stub accept/reject; toPkAddress vs G/2G/3G. |
| `test/ZkqesVerifier.fuzz.t.sol`            | toPkAddress vs `vm.addr(priv)` over 18 pinned secp256k1 vectors (true Sol-side fuzz isn't possible — see §10). |
| `test/ZkqesRegistry.admin.t.sol`           | Constructor (initial state, zero-address rejection × 3), root rotation, admin transfer, both verifier setters.  |
| `test/ZkqesRegistry.register.t.sol`        | Happy path × 2 (RSA + ECDSA dispatch), 7 revert paths (proof, root, future, age, double, declHash, unknown alg), cross-algorithm dispatch sanity (RSA accepts but ECDSA route rejects, vice versa). |
| `test/ZkqesRegistry.expire.t.sol`          | Happy path, wrong signer, tampered digest, never-bound, double-expire.                          |
| `test/ZkqesRegistry.isActiveAt.t.sol`      | Full truth table: NONE, pre-bind, ACTIVE-now, ACTIVE-far-future, EXPIRED-before/at/after expiredAt. |
| `test/ZkqesRegistry.integration.t.sol`     | **Pending Task 10** — real Groth16 verifier + real proof fixture.                                 |
| `test/helpers/SignatureHelpers.sol`      | `expireDigest(pkAddr, chainId, boundAt)` — single source of truth shared with off-chain fixture generators. |

When adding a feature: write the test first. Stub-verifier tests are
fast and exhaustive; reserve the integration suite for end-to-end
exercises with real proofs. If you introduce a new error, add one
revert test per call site that can fire it.

## 8. Gas

Current baseline (snapshots/gas-snapshot.txt):

- `register` (happy path, RSA dispatch): ~120k — well under the 400k spec budget.
- `expire` (happy path): ~50k for the diff over a fresh `register`.
- `isActiveAt`: single SLOAD plus arithmetic, view function.
- Both admin setters: ~28k for state change + event.
- Constructor: ~3.7M (generated verifier code dominates once real ones land).

CI MUST run `forge snapshot --check --snap packages/contracts/snapshots/gas-snapshot.txt`
to flag regressions. Refresh deliberately — when you make an intentional
gas-cost change, re-run `forge snapshot --snap …` and call out the delta
in the commit message.

### 8.1 V3 `register` gas target (split-proof)

`ZkqesRegistryV3.register` calls **two** Groth16 `verifyProof`s — one leaf
(13 public signals), one chain (5 public signals). BN254 pairing +
linear combination is a ~200k-per-call fixed floor on the EVM, so two
calls contribute ~400–440k just for the cryptographic work. Add stub-
path overhead (~197k measured in `test/ZkqesRegistryV3.register.t.sol`)
and the real-path projection is **~600k** per `register`.

Target: **≤ 700k** for V3 `register` happy path against real verifiers.
Anything above 700k is a hard stop — it implies something off in the
snarkjs verifier generation or our input packing. For reference,
mainnet ZK-circuit deployments sit in the same order of magnitude
(Tornado Cash deposit ~1.1M, Semaphore ~500k); 600k for a one-time,
one-per-user registration is in line with ZK conventions. At 20 gwei
that's ~$36 on mainnet / negligible on L2.

Real-path number will be pinned in `snapshots/gas-snapshot.txt` after
circuits-eng emits the ceremony `ZkqesGroth16VerifierEcdsa{Leaf,Chain}.sol`
contracts and lead pumps them into `src/verifiers/`.

## 10. Gotchas

- **EVM word is big-endian.** `bytes32(uint256)` already gives the
  big-endian byte representation of the integer. Do NOT add a
  reverse-bytes step in `toPkAddress` — earlier draft pseudocode had
  one, it's wrong, and an entire test class (`ZkqesVerifier.fuzz.t.sol`)
  exists to keep that mistake from being re-introduced.
- **`vm.addr(priv)` does not expose the underlying secp256k1 (x, y).**
  Solidity has no native curve op to derive xy from a fuzzed privkey.
  We use 18 pinned (priv, x, y) vectors generated with python's `ecdsa`
  library. For a true fuzz you'd FFI to a Rust secp256k1 helper —
  acceptable but not currently wired.
- **`forge script` writes a `broadcast/` directory.** It writes to both
  `./broadcast/` (worktree root) and `packages/contracts/broadcast/`
  depending on invocation. Both are gitignored at repo root; if you see
  one of them appear, double-check before staging.
- **`ecrecover` is used in two places.** `ZkqesVerifier.toPkAddress`
  computes `keccak256(x||y)[12:]` directly from the limb-packed pubkey
  (no signature involved); `ZkqesRegistry.expire` uses real `ecrecover`
  on a domain-separated digest (`keccak256(abi.encode("QKB_EXPIRE_V1",
  pkAddr, block.chainid, boundAt))`).
- **Stub verifier is test-only.** Never deploy `StubGroth16Verifier` to
  a chain users will rely on. The Deploy script's stub fallback exists
  solely for anvil dry-runs and CI smoke tests.
- **Submodules.** `lib/forge-std` (v1.9.4) and `lib/openzeppelin-contracts`
  (v5.1.0) are git submodules pinned in `foundry.lock` at repo root.
  Run `git submodule update --init --recursive` after a fresh checkout.
- **`fs_permissions` is restrictive on purpose.** Only
  `./packages/contracts/test/fixtures` is readable from Solidity. If you
  need a new fixture path, mirror the file under that directory rather
  than widening the permission scope.

## 11. Extending

### Adding a third QES algorithm (e.g. EdDSA, DSTU-4145)

1. Pick a new `algorithmTag` constant — append, never reuse:
   ```solidity
   uint8 internal constant ALG_EDDSA = 2;
   ```
2. Add a verifier slot + setter:
   ```solidity
   IGroth16Verifier public eddsaVerifier;
   function setEddsaVerifier(IGroth16Verifier v) external onlyAdmin { … }
   ```
3. Extend the constructor to take and validate the new verifier.
4. Add a dispatch arm in `register`:
   ```solidity
   else if (i.algorithmTag == ALG_EDDSA) v = eddsaVerifier;
   ```
5. Update `Binding` and `BindingRegistered` consumers if needed (the
   `algorithmTag` field already supports up to 255 algorithms).
6. Add admin tests (zero-address constructor revert, setter rotate +
   onlyAdmin + zero), happy path and dispatch-mismatch revert tests in
   `test/ZkqesRegistry.register.t.sol`.
7. Refresh the gas snapshot.
8. Update this CLAUDE.md and orchestration §2.0.

### Adding a new circuit-level public input

Touches all three packages; coordinate with circuits-eng and web-eng
before starting. `arr[12]` is fully consumed; an additional input
requires the snarkjs verifier to expand to `uint256[14]`, the
`IGroth16Verifier` interface to widen, every test to update, and the
ceremony to re-run.

## 12. Phase 1 explicit non-goals

For clarity when reviewing scope-creep PRs — these are deferred to
Phase 2 or later, NOT lurking bugs:

- **Historical-root rotation receipts.** `updateTrustedListRoot` overwrites
  in place; there is no on-chain log of the prior roots beyond the event
  stream. Indexers reconstruct history off-chain.
- **Per-context binding enforcement.** `ctxHash` is stored but the
  registry does not gate `isActiveAt` on a matching context. Consumers
  that care about context bind via the off-chain binding statement.
- **QEAA attribute disclosure.** The proof discloses pk, ctx, declaration,
  algorithm, timestamp, and the LOTL root inclusion — nothing else from
  the QES certificate. Selective attribute disclosure is Phase 2.
- **Escrow hooks (zkqes V3/V4).** The `escrow_commitment` field in the binding
  statement is `null` in Phase 1; on-chain escrow contracts are Phase 2.
- **Mixed RSA/ECDSA chains.** The proof attests to a single QES leaf
  cert chained to a single intermediate, both signed with the same
  algorithm. Mixed-algorithm chains are out of scope.
- **EIP-712 typed-data signatures for `expire`.** The current digest is
  a raw `keccak256(abi.encode(domain, …))` — sufficient because the
  bound key is a one-purpose key, but future versions may switch to
  EIP-712 for wallet UX. Bump `EXPIRE_DOMAIN` if you do.
- **Account abstraction integration.** `register` requires an EOA-style
  caller (anyone can submit, but the proof contents identify the bound
  key). Smart-contract-account wrapper helpers are Phase 2.

## 13. V3/V4 escrow additions

Phase 2 extends this package in three ways. None of the Phase 1 invariants
above relax; they compose.

### 13.1 14-signal public layout (restored from Phase-1 drop)

Phase 1 shipped a 12-signal split-proof fallback (spec §5.4) because the
unified ceremony hit 28 GB RAM on a dev box. Phase 2 runs the unified
ceremony on a 48+ GB local machine and restores the original layout per
zkqes spec §14.3:

```
[0..3]   pkX limbs (4 × uint64 LE)
[4..7]   pkY limbs (4 × uint64 LE)
[8]      ctxHash
[9]      rTL                  — restored from Phase-1 drop
[10]     declHash
[11]     timestamp
[12]     algorithmTag         — restored; 0 = RSA, 1 = ECDSA
[13]     nullifier            — new §14.4 primitive
```

`ZkqesVerifier.Inputs` gains `rTL`, `algorithmTag`, `nullifier`; loses
`leafSpkiCommit` (the glue-only field from the split proof). The
`IGroth16Verifier.verifyProof` input widens to `uint256[14]`.

### 13.2 Dual-verifier dispatch (restored)

`ZkqesRegistry` holds two settable verifier slots — `rsaVerifier` and
`ecdsaVerifier`. `register` (and the new `_authorizeBinding` gate used by
the escrow surface) dispatches on `i.algorithmTag`:

- `0` → `rsaVerifier`,
- `1` → `ecdsaVerifier`,
- anything else → `revert UnknownAlgorithm()`.

Admin setters are split: `setRsaVerifier`, `setEcdsaVerifier`. The single
Phase-1 `setVerifier` is gone. `VerifierUpdated` event now carries
`uint8 indexed algorithmTag` so indexers can tell the two rotation streams
apart.

### 13.3 On-chain `rTL` equality check (restored)

`register` reverts `RootMismatch` when `i.rTL != trustedListRoot`. Phase 1
couldn't do this (the leaf proof didn't carry rTL); Phase 2's unified
proof does, so the admin's LOTL-freshness duty is now enforced on-chain
rather than solely off-chain.

### 13.4 Nullifier primitive (§14.4) — amended 2026-04-18

Construction (off-chain, in the circuit). See
`docs/superpowers/specs/2026-04-18-person-nullifier-amendment.md`:

```
subjectSerialLimbs[4]  = 4 × uint64 LE packing of the zero-padded-to-32
                         bytes of the cert's subject serialNumber (OID
                         2.5.4.5), as emitted by X509SubjectSerial.circom
subjectSerialLen       = byte length of that content (1..32)

secret                 = Poseidon(subjectSerialLimbs[0..3], subjectSerialLen)
nullifier              = Poseidon(secret, ctxHash)
```

Derived from the eIDAS subject-serialNumber mandated by ETSI EN 319
412-1 §5.1.3 (format `<TYPE><CC>-<national-id>`, e.g. `PNOUA-3456789012`,
`PNODE-12345678`, `TINPL-1234567890`). Stable across cert renewals ⇒
one-person-per-`ctxHash` Sybil resistance. On-chain surface unchanged.

On-chain storage:
- `mapping(bytes32 => bool) public usedNullifiers` — uniqueness guard.
- `mapping(bytes32 => address) public nullifierToPk` — reverse lookup
  used by the Sedelmeir DB-CRL revocation event payload.
- `mapping(bytes32 => bytes32) public revokedNullifiers` — non-zero value
  = CRL-revoked with that `reasonHash`.

`register`:
- reverts `NullifierUsed` when `usedNullifiers[i.nullifier]` is set,
- on success writes both `usedNullifiers` and `nullifierToPk`,
- emits `BindingRegistered(pkAddr indexed, algorithmTag indexed, ctxHash,
  declHash, nullifier)`.

`revokeNullifier(nullifier, reasonHash)`:
- admin-only,
- `reasonHash` must be non-zero (reuses `ZeroAddress` for zero-reason
  revert),
- reverts `UnknownNullifier` if the nullifier was never registered,
- reverts `NullifierAlreadyRevoked` on double-revoke,
- emits `NullifierRevoked(nullifier indexed, pkAddr indexed, reasonHash)`.

`isActiveAt` extension: returns `false` when the mapped binding's
nullifier has a non-zero entry in `revokedNullifiers`, regardless of
`status` or `expiredAt`.

### 13.5 Escrow surface

State:
```solidity
struct EscrowEntry {
    bytes32 escrowId;    // commitment = sha256(Econfig)
    address arbitrator;  // contract implementing IArbitrator
    uint64 expiry;
    bool revoked;
}
mapping(address => EscrowEntry) public escrows;  // keyed by pkAddr
```

Escrows are keyed by `pkAddr` (the derived Ethereum address), same key
space as the Phase-1 binding table. This avoids ambiguity about which
serialisation of the raw pk bytes is canonical.

Functions:
- `registerEscrow(escrowId, arbitrator, expiry, proof, inputs)`:
    * `arbitrator != 0`, `expiry > block.timestamp`, no prior escrow;
    * runs `_authorizeBinding` (dispatch + rTL + verify + pk derivation +
      `bindings[pkAddr].status == ACTIVE`);
    * emits `EscrowRegistered(pkAddr indexed, escrowId indexed, arbitrator, expiry)`.
- `revokeEscrow(reasonHash, proof, inputs)`:
    * same auth gate;
    * requires `e.escrowId != 0` and `!e.revoked`;
    * sets `e.revoked = true`;
    * emits `EscrowRevoked(pkAddr indexed, escrowId indexed, reasonHash)`.
- `escrowCommitment(pkAddr)` view — returns the escrowId iff active, else
  `bytes32(0)`.
- `isEscrowActive(pkAddr)` view — boolean.

Error set added: `EscrowExists`, `NoEscrow`, `EscrowAlreadyRevoked`,
`EscrowExpiryInPast`, plus `NullifierUsed`, `NullifierAlreadyRevoked`,
`UnknownNullifier`, `UnknownAlgorithm`, `RootMismatch` (the latter three
restore Phase-1-era semantics that the split-proof fallback had dropped).

### 13.6 Design note — no circuit-level escrowId

We deliberately do NOT add `escrowId` as a public signal. The Phase-1
pk-ownership proof is sufficient authentication: "valid proof for pkAddr
⇒ Holder, ⇒ authorised to touch the Holder's own escrow". Threading
escrowId through any circuit would force a fresh witness (and possibly a
new ceremony if the circuit spec widens) per escrow operation. Spec §14
+ orchestration §2.3 both commit to this.

### 13.7 Arbitrator event contract

`src/arbitrators/IArbitrator.sol` freezes:

```solidity
event Unlock(bytes32 indexed escrowId, bytes recipientHybridPk);
```

The off-chain zkqes unlock evaluator subscribes to this event across all
registered arbitrator contracts. `test/ArbitratorEvent.t.sol` asserts
`keccak256("Unlock(bytes32,bytes)") == IArbitrator.Unlock.selector` so
any drift in the event shape breaks the build before shipping.

Two concrete arbitrators:
- `AuthorityArbitrator(authority)` — ecrecover-gated, one-shot per
  `evidenceHash`. Digest: `keccak256(abi.encode(escrowId,
  recipientHybridPk, evidenceHash))`.
- `TimelockArbitrator(holderPing, timeoutSeconds)` — dead-man switch.
  `ping()` from `holderPing` resets the timer. Anyone may call
  `requestUnlock(escrowId, recipientHybridPk)` once
  `block.timestamp >= lastPing + timeoutSeconds`. Per-`escrowId` one-shot.

## 14. V5 — Base-targeted single-circuit registry (`feat/v5arch-contracts`)

V5 is a clean re-architecture, not a Phase-1 → Phase-2 extension. It
trades the V4 split-proof scheme (separate leaf + chain Groth16 proofs
with on-chain SpkiCommit equality glue) for a **single-circuit** proof
that commits to the leaf SPKI, the intermediate SPKI, and the CMS
binding signature in one go. The on-chain ECDSA work moves to the
EIP-7212 (RIP-7212) P-256 precompile at `0x0000…0100`, which is live
on Base mainnet + Base Sepolia + the OP-stack sister chains.

### 14.1 Frozen interface contracts (orchestration §2 / spec §0)

**14-element public-signal layout** (`uint[14] input`) — index drift
breaks integration silently. snarkjs auto-generated verifiers consume
exactly this shape:

```
[0]  msgSender              ≤2^160 (uint160(holder))
[1]  timestamp              ≤2^64
[2]  nullifier              Poseidon₂ output
[3]  ctxHashHi              uint128 (top 16 bytes of the 32-byte hash)
[4]  ctxHashLo              uint128 (bottom 16 bytes)
[5]  bindingHashHi          uint128
[6]  bindingHashLo          uint128
[7]  signedAttrsHashHi      uint128
[8]  signedAttrsHashLo      uint128
[9]  leafTbsHashHi          uint128
[10] leafTbsHashLo          uint128
[11] policyLeafHash         uint256 (already field-domain Poseidon)
[12] leafSpkiCommit         uint256 (= SpkiCommit(leafSpki))
[13] intSpkiCommit          uint256 (= SpkiCommit(intSpki))
```

`SpkiCommit(spki)`:
1. Parse 91-byte named-curve P-256 DER SPKI → X (32 bytes), Y (32 bytes).
2. Decompose each coord into 6 × 43-bit LE limbs.
3. `SpkiCommit = Poseidon₂(Poseidon₆(X_limbs), Poseidon₆(Y_limbs))`.

iden3 / circomlib BN254 Poseidon parameters. **Byte-equivalent** across
circuits-eng's TS (`spki-commit-ref.ts`), our Solidity port
(`P256Verify.spkiCommit`), and flattener-eng's TS port. The §9.1 parity
gate enforces this — see `test/SpkiCommitParity.t.sol`.

`IZkqesRegistry` ABI is preserved verbatim across V4↔V5 (`isVerified`,
`nullifierOf`, `trustedListRoot`). Third-party SDK consumers + the V4
`ZkqesCertificate` contract bind unchanged. See `ZkqesCertificate.v5.t.sol`
for the cross-version compat test.

### 14.2 5-gate `register()` body (V5 §6)

`register()` runs five gates in fixed order. Each has a dedicated custom
error for diagnostics; gate ordering is deliberate (closer-to-proof
gates run first, so a tampered signal hits Gate 2a before Gate 3).

| # | Gate          | Check                                                     | Errors                              |
|---|---------------|-----------------------------------------------------------|-------------------------------------|
| 1 | Groth16       | snarkjs verifier accepts proof + 14 public signals        | `BadProof`                          |
| 2a| Calldata bind | sha256(signedAttrs) hi/lo, SpkiCommit(leaf), SpkiCommit(int) match the public signals | `BadSignedAttrsHi/Lo`, `BadLeafSpki`, `BadIntSpki` |
| 2b| 2× P-256      | EIP-7212 verifies leaf sig over signedAttrs, intermediate sig over leafTbsHash | `BadLeafSig`, `BadIntSig` |
| 3 | Trust Merkle  | `bytes32(intSpkiCommit)` ∈ trustedListRoot (depth 16)     | `BadTrustList`                      |
| 4 | Policy Merkle | `bytes32(policyLeafHash)` ∈ policyRoot (depth 16)         | `BadPolicy`                         |
| 5 | Timing/sender/replay | `block.timestamp - sig.timestamp ≤ MAX_BINDING_AGE` (1h), `sig.msgSender == uint160(msg.sender)`, no prior nullifier from this wallet | `FutureBinding`, `StaleBinding`, `BadSender`, `AlreadyRegistered` |

After Gate 5 passes, the registry writes:
- `nullifierOf[msg.sender] = bytes32(sig.nullifier)` (V5.1: write-once on first-claim only)
- emits `Registered(holder, nullifier, timestamp)`.

V5.1 amendment migrated cross-wallet anti-Sybil from `registrantOf[nullifier]`
(now dropped) to `usedCtx[fingerprint][ctxKey]` — see the V5.1 plan at
`docs/superpowers/plans/2026-04-30-wallet-bound-nullifier-contracts.md` and
the new mappings on ZkqesRegistryV5: `identityCommitments`, `identityWallets`,
`usedCtx`. Task 2 of that plan rewrites Gate 5 with first-claim/repeat-claim
branching; Task 3 adds the `rotateWallet()` entry point.

### 14.4 V5-specific tooling caveats

- **Forge revm doesn't ship RIP-7212.** The §2.1 P256PrecompileSmoke
  test runs against revm and stays RED until Foundry catches up. Real-
  chain reachability is empirically validated by
  `script/probe-eip7212.ts` (4/4 OP-stack chains return 0x…01 + spec-
  conformant invalid-sig rejection). Unit tests for `register()`'s Gate
  2b use `vm.mockCall` on `0x100` to simulate `{accept, reject}`
  responses — see `_mockP256AcceptAll` / `_mockP256RejectAll` helpers
  in `ZkqesRegistryV5.register.t.sol`.

- **Poseidon under via_ir.** Pure-Solidity Poseidon implementations
  (vimwitch, our in-tree attempt) trip Yul's stack analyser when the
  round-helper functions get inlined. Our V4 snarkjs verifier requires
  via_ir, so we can't disable it globally. V5 uses bytecode-deployed
  Poseidon emitted by `circomlibjs/src/poseidon_gencontract.js`,
  CREATE-deployed in the registry's constructor + cached as immutables.
  Reproducibility gate: `script/check-poseidon-reproducibility.ts`
  asserts `src/libs/PoseidonBytecode.sol` matches the generator's
  output byte-for-byte.

  - **Reproducibility-check enforcement (audit anchor for Path B').**
    Because `PoseidonBytecode.sol` is opaque hex, the generator script +
    the reproducibility check together ARE the audit surface for
    Poseidon. Concrete enforcement model:

    ```bash
    # Canonical (matches the forge test / forge build pattern):
    pnpm --filter @zkqes/contracts check:poseidon

    # Equivalent direct invocation (also works from repo root):
    pnpm exec tsx packages/contracts/script/check-poseidon-reproducibility.ts
    ```

    Exit 0 = on-disk file matches the generator's stdout byte-for-byte;
    exit 1 = drift. Run before any commit that touches
    `src/libs/PoseidonBytecode.sol` or `script/generate-poseidon-bytecode.ts`.

    No GitHub Actions CI workflow runs this today — by design. Per the
    repo-root `CLAUDE.md` §"CI / verification", the worker review loop
    substitutes for general-purpose CI (lead runs `forge test` +
    inspects diff per worker commit). Two layers enforce the repro
    check today:

    1. **Worker discipline.** Whoever edits `PoseidonBytecode.sol` or
       the generator must run the script locally and confirm exit 0
       before committing.
    2. **Lead pre-merge gate.** Lead runs the script before any
       `feat/v5arch-contracts` → `main` merge. Non-zero exit blocks
       the merge.

    When the repo eventually adopts a general-purpose CI workflow
    (`.github/workflows/ci.yml` covering all four packages — likely
    post-A1, possibly during A3 audit prep), this check should be
    wired in as a pre-`forge test` step in the contracts-package job.
    Until then, this paragraph IS the merge-gate documentation.

- **node:crypto.sign(null, X) gotcha.** It signs over sha256(X), NOT X.
  See `gen-eip7212-sentinel.ts` top docblock for the full root-cause +
  the belt-and-suspenders self-verify pair that catches re-introduction.

### 14.7 V5 specific non-goals

For clarity when reviewing scope-creep PRs:

- **Upgradeability.** V4 → V5 is a fresh deploy (`DeployV5.s.sol`), not
  an upgrade. Holders re-register; relying parties dual-lookup against
  both V4 and V5 during migration. Per the V4 Phase-2 storage warning
  (§13.8) — V5's storage layout is also non-upgradeable.

- **Non-OP-stack chains.** RIP-7212 isn't live on Polygon zkEVM (per
  `probe-eip7212.ts`); V5 is targeted at Base mainnet (with Base Sepolia
  as testnet). Adding non-OP-stack support requires either revisiting
  the Daimo P256Verifier fallback (gas budget +600-700K) or waiting for
  RIP-7212 to spread further.

- **Multi-curve / multi-algorithm.** V5 is P-256 / ECDSA only. Mixed-
  algorithm chains (RSA leaf + ECDSA intermediate, etc.) are out of
  scope. The Groth16 circuit encodes the ECDSA assumption.
